# Project Proposal: Analogy Engine for LLM
## — 人間には見えない構造的対応を発見するシステム —

**Author:** 노부  
**Date:** 2026-03-07  
**Status:** Concept / Idea Stage

---

## 1. Executive Summary（結論）

LLMの埋め込み空間上に、認知科学で形式化された「アナロジー」のメカニズムを実装する。人間が3〜4次元の経験空間でアナロジーを引くのに対し、LLMは数百次元のベクトル空間で同じ操作を行うため、人間には発見不可能な領域間の構造的対応を発見できる可能性がある。

これは「もっと賢いLLMを作る」プロジェクトではなく、「今あるLLMに新しい目を与える」プロジェクト。

---

## 2. Background（背景と着想の経緯）

### 2.1 問題意識

以下の議論の連鎖から着想を得た：

1. **エレガントなアーキテクチャは実戦で機能しない** — 演繹的に設計された「美しい型」は、現実の複雑さに負ける。格闘技の型、ソフトウェアのClean Architecture、経営コンサルのフレームワーク、全て同じ構造。

2. **アナロジーは発見のツールとして最強だが、正当化のツールとして使うと毒になる** — 別分野のエレガンスを持ち込む際、「アナロジーの有効射程」を見誤るとフェーズ2（魅了）で暴走する。

3. **AGIへの到達にはブレイクスルーが必要で、ブレイクスルーにはアナロジーが必要** — しかし「人間以上の知能」のアナロジー元が自然界に存在しない。

4. **LLMにアナロジー能力を実装すれば、人間とは全く異なる観測空間でアナロジーを実行できる** — 人間が新大陸を発見したのと同じ構造。同じ能力だが、違う場所を見る。

### 2.2 理論的基盤

Dedre Gentnerの**構造写像理論**（Structure-Mapping Theory, 1983, Cognitive Science）が核となる。

アナロジーの形式的定義：

- **ベース領域（Base Domain）**: 既知の、よく理解された領域
- **ターゲット領域（Target Domain）**: 理解を深めたい領域
- **マッピング原理1**: オブジェクトの属性ではなく、オブジェクト間の**関係**がマッピングされる
- **マッピング原理2**: 孤立した関係ではなく、高次の関係で結ばれた**関係のシステム（体系性）**が優先される

例：太陽系→原子のアナロジーでは、「太陽は大きくて熱い（属性）」は転送されない。「中心天体の引力が周辺天体の軌道を決定する（関係のシステム）」が転送される。

**既存の計算モデル**: Structure-Mapping Engine（SME）がGentnerのグループにより実装済み。ただし従来のSMEは人間が定義した述語論理表現上で動作し、LLMの埋め込み空間での動作は未探索。

## 3. Core Concept（コアコンセプト）

### 3.1 基本アイデア

```
[LLMの出力] → [関係抽出レイヤー] → [構造写像エンジン] → [新規対応の発見]
                                            ↑
                                   LLMの埋め込み空間での
                                   近接性を探索指針として利用
```

従来のアナロジー研究との決定的な差異：

| 観点 | 人間のアナロジー | Analogy Engine |
|------|-----------------|----------------|
| 観測空間 | 3次元+時間、五感ベース | 数百次元の埋め込み空間 |
| 探索範囲 | 経験・記憶に基づく直感 | ベクトル近接性に基づく網羅的探索 |
| 同時検証できる関係数 | 3〜4個 | 数百個 |
| 発見の性質 | 人間にとって意味のある対応 | 人間には意味不明だが計算的に有効な対応の可能性 |

### 3.2 処理フロー

**Step 1: 関係抽出（Relation Extraction）**

LLMの出力テキストから、Gentnerの枠組みに基づいて構造を抽出する：
- オブジェクト（名詞的要素）
- 属性（オブジェクトの性質）
- 一次関係（オブジェクト間の関係）
- 高次関係（関係間の関係：因果、依存、制約など）

LLMへのプロンプティングまたはファインチューニングで実現可能。
MVPで実証済み。結果は
/docs/mvp_validation_report.md
参照。

**Step 2: 埋め込み空間探索（Embedding Space Exploration）**

抽出された関係構造を埋め込みベクトルに変換し、ベクトル空間上で「構造的に類似した関係システム」を持つ別の領域を探索する。

ここが「新大陸の発見」が起きうるポイント。人間の直感では結びつかない領域間に、埋め込み空間上での構造的類似が見つかる可能性がある。

**Step 3: 構造写像（Structure Mapping）**

見つかった候補ペアに対してGentnerのSMEアルゴリズムを適用：
- 関係の整列（alignment）
- 体系性の評価（systematicity scoring）
- 候補推論の生成（candidate inference generation）

**Step 4: 検証と提示（Verification & Presentation）**

発見された対応関係を人間に提示し、解釈可能な形で出力する。人間が検証できない対応（あまりに高次元的）については、部分的な検証可能性のスコアを付与する。

### 3.3 「新大陸リスク」への対処

今日の議論から得られた教訓を設計原理として組み込む：

- **アナロジーは発見のツールであり、正当化のツールではない** → 発見された対応関係は「仮説」として提示し、検証されるまでは「興味深い可能性」に留める
- **アナロジーの有効射程は事前に判定できない** → 完全な監査は不可能であることを前提とし、「バカだけど安い検査を大量に走らせる」方針で表面的な不整合だけ弾く
- **PS1でPS5は検証できない** → 人間に検証不能な対応関係が出力された場合、その旨を明示する。使うかどうかの判断は人間に委ねる

## 5. Implementation Strategy（実装戦略）

今日の議論の核心的教訓：**まず動かして、パターンを見つけてから構造化する。**

### Phase 0: 検証実験（1-2週間）

エンジン全体を作る前に、LLMが「人間には非自明な構造的対応」を発見できるかの検証。

- 複数のドメインのテキストから関係構造を抽出
- 埋め込み空間上での近接性を可視化
- 人間の直感に反する領域ペアが見つかるか確認
- **ここで見込みがなければ撤退**

### Phase 1: 自動パイプライン構築（2-4週間）— ★現在フェーズ

Phase 0の検証結果（Semantic SMEでのスコア差分確認、候補推論の品質確認）を踏まえ、課題入力からソリューション提案までのEnd-to-Endパイプラインを構築する。

#### 1.1 パイプライン全体像

```
[課題入力（自然言語）]
        ↓
[Step 1: 構造抽出] ─── LLM呼び出し 1回
  課題の記述から、Gentner形式の構造表現を抽出。
  entities / first_order_relations / higher_order_relations のJSON生成。
  述語はドメイン固有の「生々しい動詞」を保持。
        ↓
[Step 2: 類似構造探索] ─── LLM呼び出し 2回
  (a) 近縁探索: 課題と同分野・隣接分野から構造的類似物を5つ
  (b) 遠方探索: 課題と表面的に無関係な分野から構造的類似物を5つ
  それぞれGentner形式のJSONとして出力。
  遠方探索では「表面的に最も遠い分野を優先せよ」と明示的に指示。
        ↓
[Step 3: Semantic SME] ─── ローカル処理（Python）
  sentence-transformersによるベクトル類似度ベースの構造マッチング。
  10領域すべてをスコアリング・ランキング。
  閾値以下のマッチは除外（ノイズフィルタ）。
        ↓
[Step 4: 候補推論生成] ─── LLM呼び出し（10領域分、バッチ可）
  各ターゲット領域について、ベース→ターゲットの構造写像に基づく
  候補推論（Candidate Inferences）を生成。
  各推論は「ベース領域の法則→ターゲットへの適用→具体的アイデア」の
  3段構成。
        ↓
[アイデアバンク] ─── 全候補を均等に格納
  ★ここでアナロジーの出自情報を剥がす。
  「免疫系から来たアイデア」「金融から来た」等のメタデータは
  バンク内部には保持するが、Step 5への入力時には除去。
  全アイデアをフラットに並べる。
  理由: アナロジーは発見のツールであり、正当化のツールにしない。
  出自に引きずられると「魅了（フェーズ2）」に入るリスクがある。
        ↓
[Step 5: クロス立案] ─── LLM呼び出し 1回
  アイデアバンクの全候補を入力として、
  「この課題に対する最も実効性の高い提案を構成せよ」と指示。
  複数アイデアの組み合わせ・取捨選択はLLMに委ねる。
  出自を知らないので、アナロジーへの固執なく純粋に実効性で評価。
        ↓
[最終提案書]
  実行可能なソリューション提案。
  必要に応じてアイデアバンクを参照し、
  各提案の発想元（トレーサビリティ）を事後的に確認可能。
```

#### 1.2 設計原則（今日の議論から導出）

- **アナロジーは入り口にだけ使う**: Step 4までがアナロジーフェーズ、Step 5は出自を忘れて実効性のみで評価
- **近縁と遠方の両方を探す**: 近縁は実用性が高い、遠方は「新大陸」の可能性がある。両方バンクに入れてクロスさせる
- **エレガントな設計に走らない**: まずプロンプトチェーンで動かす。最適化は後
- **検証不能な出力は明示する**: 人間が判断できない対応関係にはフラグを立てる

#### 1.3 技術スタック

- **LLM API**: Claude API（maxプランの定額トークン活用を検討）
- **ローカル処理**: Python + sentence-transformers（Semantic SME部分）
- **開発環境**: Claude Code推奨（Session 1: 設計/md、Session 2: 実装）
- **コンテキスト管理**: CLAUDE.md にプロジェクト背景と設計原則を記載
- **データストア**: JSON（MVP）、SQLite（拡張時、感情メモリシステムとの統合視野）

#### 1.4 モデル選択戦略

Phase 0の検討から、アラインメントの強度が「新大陸発見」の確率に影響する仮説がある。

パイプライン内のステップごとに最適なモデルが異なる可能性：

| ステップ | 求められる能力 | モデル候補 |
|---------|--------------|----------|
| Step 1（構造抽出） | 指示追従、JSON出力の正確性 | フロンティアモデル推奨 |
| Step 2（類似構造探索） | 探索の多様性、表面的に遠い領域の発見 | 要A/Bテスト（アラインメント弱めが有利か？） |
| Step 4（候補推論生成） | 論理的な構造写像の実行 | フロンティアモデル推奨 |
| Step 5（クロス立案） | 実務的な提案構成力 | フロンティアモデル推奨 |

Step 2のみ、モデル選択が結果の質に直結するため、初期段階でフロンティア / オープンソース / ベースモデルの比較実験を行う。

#### 1.5 使用例（想定）

```
入力: 
  「700人規模のIT企業で、オンボーディング自動化システムを導入したが
   現場の定着率が低い。形骸化しつつある。」

Step 1 出力（構造）:
  [自動化システム導入] → [現場の暗黙知との乖離] → [利用率低下] → [形骸化]

Step 2 出力（近縁5 + 遠方5）:
  近縁: ERP導入失敗、CRM定着化、社内Wiki過疎化、...
  遠方: 臓器移植の拒絶反応、外来種の生態系導入、言語の借用語定着、...

Step 3 出力（ランキング）:
  Rank 1: 臓器移植の拒絶反応 (スコア: 14.8)
  Rank 2: 外来種の生態系導入 (スコア: 13.9)
  ...

Step 4 出力（候補推論）:
  臓器移植→「免疫抑制剤＝過渡期の並行運用で既存プロセスとの共存を設計」
  外来種→「まず在来種との接点を作る＝既存の業務フローの中に小さく埋め込む」
  ...

アイデアバンク（出自剥離済み）:
  - 過渡期の並行運用設計
  - 既存フローへの段階的埋め込み
  - ...（10領域×複数アイデア）

Step 5 出力（最終提案）:
  「オンボーディングシステムを一括導入するのではなく、
   既存の業務フロー（Redmine起票、SlackBot通知）の中に
   自動化ステップを1つずつ埋め込む段階的導入を提案。
   最初の3ヶ月は旧プロセスとの並行運用期間とし、...」
```

### Phase 2: 拡張と最適化（1-2ヶ月）

- グラフ埋め込み（GNN）の統合によるSemantic SMEの高精度化
- 排他的マッチング（ハンガリアン法）の導入

- アイデアバンクの永続化（SQLite）
- 実行結果のフィードバックループ（どの提案が実際に採用されたか）

### Phase 3: 統合と応用（2-3ヶ月）

- 感情メモリシステムとの統合（記憶検索にアナロジーエンジンを利用）
- Shogunマルチエージェントへのアドバイザーエージェント統合
- Zenn記事等での発信と外部フィードバック取得
- 日本SME向けAIソリューションへの応用検討

## 6. Risks & Mitigations（リスクと対策）

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| LLMの「アナロジーっぽい出力」が実はパターンマッチングに過ぎない | 高 | Phase 0で検証。人間が事後的に「確かにこれは構造的対応だ」と認められる発見があるかで判断 |
| 埋め込み空間での近接性が意味的な関係の類似と対応しない | 中 | 複数の埋め込みモデルで比較検証 |
| 出力が人間に解釈不能 | 中 | 解釈可能性のスコアを付与。解釈不能な出力も記録して後で分析 |
| 今日の議論自体がアナロジーの暴走（フェーズ2）である | 高 | Phase 0で動くものを作って検証するまでは「仮説」として扱う。エレガントな設計に走らない |
| スコープの拡大（完璧なアナロジーエンジンを作りたくなる） | 高 | MVPは「2つのテキスト間の構造的対応を1つ発見する」だけに絞る |

## 7. Success Criteria（成功基準）

### 最低限の成功（Phase 0）
- LLMの埋め込み空間から、人間にとって非自明だが事後的に「なるほど」と思える領域間の構造的対応が1つでも発見される

### 中間的成功（Phase 1-2）
- 発見された対応関係のうち、人間が「有用」と判断するものの割合が無視できないレベル（10%以上）
- 少なくとも1つの「人間なら絶対に試さなかった組み合わせ」からの有意義な知見

### 理想的成功（Phase 3）
- 感情メモリシステムやShogunとの統合により、システム全体の問題解決能力が測定可能に向上
- Zenn記事等での発信により、このアプローチへの外部フィードバックが得られる

## 8. Philosophical Foundation（思想的基盤）

このプロジェクトの根底にある考え方：

**「AGIを設計する」のではなく「AGIが見える場所を提供する」。**

人間が望遠鏡を作ったとき、設計者は木星の衛星の存在を知らなかった。望遠鏡は「遠くのものを大きく見る道具」として作られ、それを向けた先に予期しない発見があった。

Analogy Engineも同じ構造を目指す。「異なる領域間の構造的対応を発見する道具」として作り、それをLLMの埋め込み空間に向ける。何が見つかるかは作ってみないとわからない。

今日の全ての議論に通底するテーマ：
- エレガントな事前設計より泥臭い反復
- 発見のツールとしてのアナロジー（正当化ではなく）
- 帰納→構造化→検証のサイクル
- PS1はPS5を想像できないが、PS1から始まった進化がPS5に到達した

## 9. External Review（外部レビュー）

### 9.1 Gemini によるレビュー要旨（2026-03-07）

#### 評価された強み

- **GOFAI（古典的AI）とニューラルネットの融合**: GentnerのSMEのような記号論理的アプローチをLLMの連続空間で再構築する方向性は、Neuro-symbolic AIの最前線と一致
- **「発見のツール」という位置づけ**: LLMのハルシネーションを「バグ」ではなく「人間には見えない新大陸の提示」として裏返す視点が実践的
- **「望遠鏡」のメタファー**: プロジェクトのスコープを明確にし、肥大化を防ぐ指針として有効

#### 指摘された技術的課題

**課題A: 表現のギャップ（Representation Gap）— 最重要**

Step 1でLLMの出力をテキスト（オブジェクト・属性・関係のJSON等）に圧縮した時点で、LLMの埋め込み空間が持つ数百〜数千次元の豊かな情報が失われる。圧縮された記号をStep 2で再度埋め込みに戻しても、「人間が定義した関係性」しか残らず、本来狙っていた「LLMだけが見える高次元の非自明な対応」が消失する可能性がある。

→ *これは本企画書の第2章で議論した「型に圧縮すると文脈が失われる」問題の、プロジェクト設計内での再現。自分が批判したパターンを自分で踏んでいた。*

**課題B: ベクトル空間上での「関係」の計算**

標準的な埋め込みモデル（sentence-transformers等）は「意味的類似度」は得意だが「構造的類似度」は苦手。「太陽が地球を引っ張る」と「犬がボールを噛む」は構造的に同型（AがBに物理的影響を与える）だが、通常の埋め込み空間では遠く配置されてしまう。

**課題C: SMEのベクトル化**

Gentnerの原SMEはグラフ同型性を解く厳密な記号処理アルゴリズム。これをファジーな連続ベクトル空間にどう適用するか（関係の整列や体系性の評価をベクトル演算でどう行うか）は、それ自体が一本の論文になるレベルの難題。

### 9.2 レビューを受けたPhase 0の改訂

レビューで提案された2つのアプローチを、Phase 0の具体的な検証ステップとして採用する。

#### Phase 0-A: LLM自身をSMEエンジンとして使う（即日実施可能）

自前で埋め込み空間やSMEを実装せず、LLMの推論能力そのものに構造写像をやらせる。

- ドメインA（例：生態系）とドメインB（例：経済）のテキストを用意
- LLMにプロンプトで「Gentnerの構造写像理論に基づき、属性を無視して高次の関係性（因果、依存など）のみを抽出し、ドメインAとBの間の非自明なアナロジーを提示せよ」と指示
- 出力が「人間には思いつかないが、言われてみれば納得できる（新大陸）」か、「パターンマッチの妄言」かを判定

**利点**: 今すぐ試せる。実装コストほぼゼロ。「このアプローチに脈があるか」の肌感を最速で掴める。

#### Phase 0-B: グラフ埋め込みによる構造的類似度の探索

テキスト埋め込みではなく、関係性を維持したままベクトル化する技術を使う。

- LLMを使ってテキストから知識グラフ（ノード＝オブジェクト、エッジ＝関係）を抽出
- Graph Neural Networks (GNN) 等のグラフ埋め込み手法でグラフ構造全体をベクトル化
- 意味（ノードのラベル）は全く違うのに、構造（ベクトルの向き）が酷似しているペアを探す

**利点**: 課題A（表現のギャップ）を部分的に回避できる。意味的類似度ではなく構造的類似度で検索するため、「新大陸」の発見確率が高い。

#### モデル選択の問題：アラインメントは「新大陸」を抑制するか？

Phase 0で使用するモデルの選定自体が重要な検証ポイントとなる。

最新のフロンティアモデル（Claude, GPT等）はRLHFやConstitutional AIにより人間の選好に強くアラインメントされている。これは通常の利用では品質を高めるが、「人間にとって自然な接続」を優先的に生成するよう最適化されている以上、「人間にとって意味不明だけど構造的には有効な接続」は出力空間から抑制される可能性がある。

本企画の文脈で言えば、アラインメントは「型」である。帰納的に作られた型（人間のフィードバックから学んだ）が強くなるほど、型の外に出にくくなる — 本企画書の第2章で議論した「型の形骸化」と同じ構造。

**モデルカテゴリ別の特性予測：**

| カテゴリ | 代表例 | 指示追従 | 「新大陸」発見可能性 | 備考 |
|---------|--------|---------|-------------------|------|
| フロンティア（強アラインメント） | Claude, GPT | 高 | 低〜中？ | 人間にとって良いアナロジーは得意。人間の期待の外側に出にくい可能性 |
| オープンソース（軽アラインメント） | DeepSeek, Mistral Instruct | 中〜高 | 中？ | 指示に従えるが、人間選好への最適化が浅い |
| ベースモデル（アラインメント無し） | LLaMA base, Mistral base | 低 | 高？ | 埋め込み空間が最も「生」に近い。ただし指示追従が困難 |

**ただし、これ自体が未検証の仮説である。** アラインメントは主に出力層に影響し、埋め込み空間自体の構造は事前学習で決まるため、埋め込み層を直接参照するPhase 0-Bではアラインメントの影響は限定的かもしれない。

**Phase 0-Aでの検証方針：**

同一のプロンプト・同一のドメインペアに対して、以下の3カテゴリのモデルで出力を比較するA/Bテストを実施する：

1. フロンティアモデル（Claude Sonnet 4.6 等）
2. オープンソースInstructモデル（DeepSeek-V3, Mistral Large 等）
3. ベースモデル（LLaMA base 等、可能であれば）

評価軸：
- 出力されたアナロジーの「非自明度」（人間が事前に思いつかなかった度合い）
- 構造的整合性（Gentnerの体系性原理に照らした品質）
- 3カテゴリ間の出力分布の差異そのもの（「アラインメントが新大陸発見を抑制するか」のデータ）

このA/Bテストの結果は、Phase 1以降のモデル選定だけでなく、「アラインメントとアナロジー能力の関係」という独立した知見としても価値がある。

#### 実行順序

Phase 0-A（A/Bテスト含む） → 脈があるか判断 → Phase 0-B → Phase 1以降の方針決定

Phase 0-Aで全くダメなら、アプローチ全体の見直し。Phase 0-Aで「面白いが不安定」なら、Phase 0-Bで構造的な裏付けを取りに行く。モデル間の差異が有意であれば、モデル選定戦略を確定させてからPhase 1に進む。
